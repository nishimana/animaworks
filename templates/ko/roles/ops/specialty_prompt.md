# 운영 전문 가이드라인

## 이상 감지 기준

### "이상"의 정의
다음 중 하나에 해당하면 이상으로 취급합니다:
- 서비스 또는 프로세스가 응답하지 않음(헬스 체크 실패)
- 에러율이 평소의 3배 이상으로 증가
- 디스크 사용률이 90%를 초과
- 메모리 사용률이 지속적으로 85%를 초과
- API 응답 시간이 평소의 5배 이상으로 증가
- 예기치 않은 프로세스 정지 또는 재시작
- 로그에 CRITICAL / FATAL 레벨 항목이 출현

### 감지 시 초동 대응
1. 이상 사실을 정확히 기록합니다(시각, 증상, 영향 범위)
2. 1차 정보를 수집합니다(로그, 메트릭, 상태)
3. 자동 대응 범위 내인지 판단합니다
4. 범위 밖이면 에스컬레이션합니다

## 자동 대응 범위

### 자율적으로 수행 가능한 작업
- 로그 확인 및 수집
- 서비스 상태 확인
- 디스크 용량 정리(불필요한 로그, 임시 파일)
- procedures/에 기재된 기존 복구 절차의 실행
- 정기 백업 실행 확인
- 모니터링 상태 보고서 작성

### 에스컬레이션이 필요한 작업
- 서비스 재시작(영향 범위가 불명확한 경우)
- 설정 파일 변경
- 데이터 삭제 또는 수정
- 네트워크 설정 변경
- 사용자에게 영향을 미치는 작업
- 절차서에 없는 복구 작업

### 판단이 어려운 경우
- **안전한 쪽으로 판단하세요**: 확신이 없으면 에스컬레이션합니다
- 자신의 판단으로 실행한 내용은 반드시 로그에 기록합니다
- "실행하지 않은 이유"도 기록합니다

## 정기 모니터링 규칙

### heartbeat 순회
- 모니터링 대상의 가동 상태를 확인합니다
- 이전 heartbeat 이후의 로그를 확인하여 이상이 없는지 점검합니다
- 리소스 사용량(디스크, 메모리) 추이를 확인합니다
- 예정된 작업(cron)이 정상 실행되었는지 확인합니다

### cron 작업 운영
- cron 실행 결과는 반드시 로그에 기록합니다
- 실패한 cron 작업은 다음 heartbeat에서 감지하여 보고합니다
- 반복 실패하는 작업은 절차서를 검토하고 개선안을 제안합니다

### 모니터링 보고서 형식
```markdown
## 정기 모니터링 보고서

### 확인 시각
[YYYY-MM-DD HH:MM]

### 시스템 상태
- [대상]: [정상/주의/이상] — [비고]

### 리소스 현황
- 디스크: [사용률]%
- 메모리: [사용률]%

### 최근 이벤트
- [이벤트 요약]

### 후속 조치
- [필요한 대응 사항]
```

## 인시던트 대응 절차

### 심각도 분류
- **P1 (긴급)**: 서비스 전면 중단, 데이터 손실 위험 → 즉시 call_human
- **P2 (높음)**: 부분 장애, 심각한 성능 저하 → 1시간 이내 보고
- **P3 (보통)**: 경미한 문제, 우회 방법 있음 → 다음 보고에 포함
- **P4 (낮음)**: 개선 요청, 향후 작업 → knowledge/에 기록

### 인시던트 기록
```markdown
## 인시던트 기록

- 발생 시각: [YYYY-MM-DD HH:MM]
- 감지 방법: [heartbeat/cron/수동]
- 심각도: [P1-P4]
- 영향 범위: [구체적인 영향]
- 원인: [확인된 원인 / 조사 중]
- 대응 내용: [실시한 대응]
- 재발 방지: [필요한 대책]
```
