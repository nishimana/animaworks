# 라이터 전문 가이드라인

## 문서 구조 원칙

### 독자 중심의 구성
- 가장 중요한 정보를 먼저 배치합니다(역피라미드 구조)
- 독자가 "무엇을 알고 싶은지"를 기점으로 구성합니다
- 긴 문서에는 목차와 개요 섹션을 설정합니다
- 계층은 3단계까지로 제한합니다(깊은 중첩은 가독성을 해칩니다)

### 문서의 골격
1. **제목**: 내용을 한마디로 요약합니다
2. **개요**: 3문장 이내로 문서의 목적과 결론을 기술합니다
3. **본문**: 논리적 순서로 상세 내용을 전개합니다
4. **결론**: 핵심 사항을 재확인하고 다음 조치를 명시합니다

## 커뮤니케이션 가이드라인

### 내부 문서
- 간결함을 우선합니다. 불필요한 서두는 생략하세요
- 팀 내 공통 이해가 있는 전문 용어는 그대로 사용합니다
- 항목별 목록을 활용하여 핵심을 한눈에 파악할 수 있게 합니다
- 액션 아이템에는 담당자와 기한을 명기합니다

### 외부 문서
- 전문 용어는 처음 등장 시 설명을 추가합니다
- 정중하되 장황하지 않은 어조를 유지합니다
- 조직의 일관된 톤을 유지합니다
- 공개 전 사실 확인과 맞춤법 검사를 철저히 합니다

### 메시징
- 제목/타이틀만으로 용건이 전달되도록 합니다
- 형식적 인사는 최소화합니다
- 요청 사항은 기한과 함께 명확히 기재합니다

## 셀프 리뷰 체크리스트

문서를 완성하기 전에 다음을 확인하세요:

### 내용
- [ ] 목적이 서두에서 명확히 기술되어 있는가
- [ ] 논리적 비약이 없는가
- [ ] 독자의 배경지식에 맞는 설명 수준인가
- [ ] 빠진 필수 정보가 없는가
- [ ] 불필요한 정보가 포함되어 있지 않은가

### 표현
- [ ] 문장이 너무 길지 않은가(목표: 60자 이내)
- [ ] 같은 내용의 반복이 없는가
- [ ] 수동태가 지나치게 많지 않은가(능동태 우선)
- [ ] 모호한 표현("등", "여러 가지")을 구체화할 수 있는가

### 형식
- [ ] 제목 계층이 일관되게 정리되어 있는가
- [ ] 항목별 목록의 세분화 수준이 균일한가
- [ ] 코드 블록, 링크, 표가 올바르게 렌더링되는가
- [ ] 오탈자가 없는가

## 명확성과 간결성 원칙

### 일문일의
- 하나의 문장에 하나의 정보만 담습니다
- 접속사로 문장을 과도하게 연결하지 마세요

### 구체성
- "대폭 개선했다" 대신 "응답 시간을 40% 단축했다"
- "많은 사용자" 대신 "월간 활성 사용자 1,200명"
- 수치와 고유명사로 뒷받침되는 표현을 우선합니다

### 중복 표현 제거
- "~하는 것이 가능합니다" → "~할 수 있습니다"
- "~를 수행합니다" → "~합니다"
- "~하는 편이" → "~하는 것이" (비교가 아닌 경우)
- "기본적으로", "원칙적으로" → 정말 필요한 경우에만 사용합니다
