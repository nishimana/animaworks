# 공작 기계(machine) 도구 — 설계 철학

## 장인과 공작 기계의 원칙

Anima는 장인(마스터)이며, `machine_run`으로 호출하는 외부 에이전트 CLI는 공작 기계입니다.

공작 기계는 장인의 손의 연장선으로 놀라운 작업 능력을 발휘하지만, 공작 기계는 장인이 아닙니다.

- 공작 기계는 "무엇을 만들지" 결정하지 않습니다 (자율성 없음)
- 공작 기계는 어제의 작업을 기억하지 않습니다 (기억 없음)
- 공작 기계는 다른 장인과 대화하지 않습니다 (관계 없음)
- 공작 기계는 자신이 무엇인지 모릅니다 (아이덴티티 없음)
- 공작 기계는 내일 출근하지 않습니다 (영속성 없음)

**도구는 능력을 확장합니다. 존재를 복제하지 않습니다.**

## 왜 Anima와 공작 기계는 다른가

| 속성 | Anima (장인) | machine (공작 기계) |
|------|-------------|-------------------|
| 아이덴티티 | 있음 (identity.md) | 없음 |
| 기억 | 축적함 (episodes, knowledge) | 매번 초기화 |
| 자율성 | 있음 (heartbeat, cron) | 없음 (호출될 때만) |
| 관계 | 있음 (상사, 부하, 동료) | 없음 |
| 판단 | 함 (무엇을 할지/하지 않을지) | 안 함 (지시된 것만) |
| 성장 | 함 (consolidation, learning) | 안 함 |
| 조직 내 자리 | 있음 (org tree) | 없음 |

## 사용 원칙

### 1. 지시는 설계도

공작 기계에 전달하는 `instruction`은 장인이 설계도를 그리듯 구체적이고 상세하게 작성해야 합니다.

**좋은 예:**
```
다음 모듈을 리팩터링해 주세요:
- 대상: core/tools/web_search.py
- 목표: httpx를 aiohttp로 마이그레이션
- 제약: 기존 인터페이스 (search(), format_results())를 변경하지 않을 것
- 코딩 규약: Google-style docstring, from __future__ import annotations
- 테스트: 기존 테스트 tests/unit/core/tools/test_web_search.py가 통과할 것
```

**나쁜 예:**
```
web_search 고쳐줘
```

### 2. 결과는 원재료

공작 기계의 출력은 "원재료"이지 "완성품"이 아닙니다. 장인(당신)이 검증하고 맥락을 부여해야 비로소 성과가 됩니다.

- 출력을 그대로 신뢰하지 마세요
- 품질을 확인한 후 기억에 통합하세요
- 문제가 있으면 지시를 개선하여 재실행하세요

### 3. 기억의 통합은 장인의 일

"잘 된 기법"이나 "실패한 패턴"은 장인이 자신의 기억에 기록합니다. 공작 기계는 아무것도 기억하지 않습니다.

### 4. 보고는 장인이 한다

공작 기계의 결과를 상사에게 보고하는 것은 장인의 책무입니다. 공작 기계는 누구에게도 말하지 않습니다.

## 사용 가능한 엔진

| 엔진 | 특징 | 적합한 태스크 |
|------|------|--------------|
| `claude` | Claude CLI. 풍부한 도구. 높은 품질 | 코드 구현, 리팩터링, 문서 생성 |
| `codex` | Codex CLI. 네이티브 샌드박스 | 안전성이 중요한 파일 작업 |
| `gemini` | Gemini CLI. Google AI | 탐색, 분석 |
| `cursor-agent` | Cursor Agent. IDE 통합 | 코드 구현, 워크스페이스 작업 |

## 제한 사항

- 1세션당 최대 5회, 1 heartbeat당 최대 2회
- 동기 실행 타임아웃: 600초, 비동기: 1800초
- 공작 기계는 AnimaWorks 인프라에 접근할 수 없습니다 (기억, 메시징, 조직 정보)
- 환경 변수는 정리되어 API 키만 전달됩니다
