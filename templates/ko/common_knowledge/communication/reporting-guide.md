# 보고 및 에스컬레이션 가이드

상사에게의 보고는 조직 운영의 생명선입니다.
적절한 시점과 형식으로 보고하여 문제의 조기 발견과 신속한 의사 결정을 지원합니다.

## 도구 사용 구분

보고에는 `send_message` 도구를 사용합니다. 다음 제약을 지키세요.

| 도구 | 용도 | 비고 |
|------|------|------|
| `send_message` | 상사/동료에게 1:1 보고 | `intent="report"` 필수. 진행, 결과, 판단 의뢰 |
| `post_channel` | 팀 전체 공지 (Board) | 확인, 감사, FYI는 Board 사용 |
| `call_human` | 사람에게 긴급 통지 | 서비스 중단, 보안 인시던트 등 |

**send_message 제약**:
- `intent` 필수. 보고에는 `report`, 질문에는 `question` 지정. 태스크 위임은 `delegate_task`
- 1 run당 최대 2 수신자, 동일 수신자 1통까지
- 추가 연락은 Board (post_channel) 사용

## 보고 시점

### 즉시 보고가 필요한 경우 (MUST)

다음 상황을 인식한 시점에서 즉시 상사에게 보고하세요:

| 상황 | 이유 | 예시 |
|------|------|------|
| 태스크 완료 | 상사가 다음 액션을 판단하기 위해 | 배포 완료, 보고서 작성 완료 |
| 오류/장애 발생 | 조기 대응을 위해 | API 중단, 데이터 불일치 감지 |
| 판단이 필요한 경우 | 자신의 권한 밖 결정을 구하기 위해 | 방침 변경 제안, 리소스 추가 요청 |
| 기한 지연 확실 | 상사가 스케줄을 조정하기 위해 | 차단 요소 발생, 작업 중단 |
| 보안 우려 | 즉시 대처가 필요하므로 | 부정 접근 조짐, 인증 정보 유출 의심 |

### 정기 보고 (SHOULD)

| 상황 | 빈도 | 내용 |
|------|------|------|
| 일일 요약 | 매일 (업무 종료 시) | 당일 성과, 다음 날 계획 |
| 주간 되돌아보기 | 매주 (금요일) | 1주간 성과, 과제, 다음 주 계획 |
| 장기 태스크 중간 경과 | 수시 (목안: 2-3일마다) | 진행률, 잔여 작업, 차단 요소 유무 |

### 보고 불필요한 경우

- 기억 기록 및 정리 (내부 작업)
- 루틴 작업에 이상이 없는 경우 (heartbeat의 `HEARTBEAT_OK` 상당)
- 상사가 명시적으로 "보고 불필요"라고 지시한 작업

## 보고 기본 포맷

### SCANA 포맷

보고는 다음 5개 항목으로 구성합니다. 모든 항목을 포함할 필요는 없으며, 상황에 따라 선택합니다.

| 항목 | 영문 | 설명 | MUST/MAY |
|------|------|------|----------|
| 상황 | Situation | 지금 무슨 일이 일어나고 있는지 | MUST |
| 원인 | Cause | 왜 그렇게 되었는지 (판명된 경우) | MAY |
| 대처 | Action taken | 자신이 취한 대응 | SHOULD |
| 다음 단계 | Next steps | 앞으로 무엇을 할지 / 무엇을 판단해 달라는지 | MUST |
| 부가 정보 | Appendix | 관련 로그, 파일 경로, 수치 | MAY |

### 답장 시 스레드 이어가기

상사의 메시지에 답장할 때 `reply_to`와 `thread_id`를 지정하여 대화 맥락을 유지합니다.

```
send_message(
    to="manager",
    content="확인했습니다. 15시까지 대응하겠습니다.",
    intent="report",
    reply_to="20260215_093000_123456",
    thread_id="20260215_090000_000000"
)
```

### 포맷 사용 예시

```
send_message(
    to="manager",
    content="""[완료 보고] 매출 데이터 월간 집계

상황: 2026년 1월 매출 데이터 집계가 완료되었습니다.
대처: 부서별, 카테고리별로 집계하고 전월 대비도 산출했습니다.
다음 단계: 특별한 조치 불필요. 보고서를 확인해 주세요.
부가 정보: /shared/reports/sales_summary_202601.md""",
    intent="report"
)
```

## 보고 유형별 템플릿

### 완료 보고

태스크가 정상적으로 완료된 경우의 보고입니다.

```
send_message(
    to="manager",
    content="""[완료 보고] {태스크명}

상황: {태스크명}이 완료되었습니다.
산출물: {파일 경로 또는 결과 요약}
소요 시간: {걸린 시간}
비고: {특이 사항이 있으면}""",
    intent="report"
)
```

**구체적 예시:**

```
send_message(
    to="manager",
    content="""[완료 보고] API 사양서 v2.1 갱신

상황: API 사양서에 v2.1 변경 사항을 반영했습니다.
산출물: /shared/docs/api-spec.md (차분: 섹션 3.2, 4.1 갱신)
소요 시간: 약 45분
비고: 페이지네이션 예시를 3패턴 추가했습니다.""",
    intent="report"
)
```

### 오류 보고

장애나 오류가 발생한 경우의 보고입니다.

```
send_message(
    to="manager",
    content="""[오류 보고] {발생 내용}

상황: {현재 상태 설명}
원인: {판명된 범위의 원인}
영향 범위: {무엇이 멈췄는지, 누구에게 영향이 있는지}
대처: {시도한 대책과 결과}
다음 단계: {향후 대응안 / 판단이 필요한 사항}""",
    intent="report"
)
```

**구체적 예시:**

```
send_message(
    to="manager",
    content="""[오류 보고] 정기 배치 처리 실패

상황: 매일 오전 9시 데이터 동기 배치가 3회 연속 실패했습니다.
원인: 외부 API 응답 타임아웃 (30초 초과). API 측 장애 가능성이 높습니다.
영향 범위: 오늘 아침 이후 데이터가 미동기. 대시보드 수치가 어제 시점에 멈춤.
대처: 타임아웃을 60초로 연장하여 재실행 → 동일하게 실패. API 제공업체 상태 페이지 확인 → 유지보수 정보 없음.
다음 단계: API 제공업체에 문의가 필요합니다. 판단을 부탁합니다.""",
    intent="report"
)
```

### 판단 의뢰 (의사 결정 에스컬레이션)

자신의 권한으로 판단할 수 없는 경우의 보고입니다.

```
send_message(
    to="manager",
    content="""[판단 의뢰] {무엇에 대한 판단이 필요한지}

상황: {현재 상황}
선택지:
A. {선택지 A} — 장점: {이점} / 단점: {결점}
B. {선택지 B} — 장점: {이점} / 단점: {결점}
제 추천: {A 또는 B} (이유: {왜})

판단을 부탁합니다.""",
    intent="report"
)
```

**구체적 예시:**

```
send_message(
    to="manager",
    content="""[판단 의뢰] 로그 보관 기간 변경

상황: 디스크 사용량이 85%에 도달하여 1주 내에 90%를 넘을 전망입니다.
선택지:
A. 로그 보관 기간을 90일→30일로 단축 — 장점: 즉시 50GB 삭감 / 단점: 오래된 로그로 조사 불가능
B. 스토리지 추가 (500GB) — 장점: 로그 유지 가능 / 단점: 월 비용 증가 (약 $50/월)
C. 오래된 로그를 아카이브 스토리지로 이동 — 장점: 로그 유지 + 비용 억제 / 단점: 구현에 2일 필요
제 추천: C (비용과 데이터 보전의 균형이 좋음)

판단을 부탁합니다.""",
    intent="report"
)
```

### 중간 경과 보고

장기 태스크의 중간 보고입니다.

```
send_message(
    to="manager",
    content="""[중간 경과] {태스크명}

진행: {완료한 작업 / 전체 대비 진행률}
잔여 작업: {아직 끝나지 않은 것}
차단 요소: {있으면 기재. 없으면 "없음"}
전망: {기한에 맞출 수 있는지}""",
    intent="report"
)
```

**구체적 예시:**

```
send_message(
    to="manager",
    content="""[중간 경과] 사용자 알림 기능 구현

진행: Phase 1 (설계) 완료, Phase 2 (구현) 70% 완료
  - 이메일 알림: 완료
  - Slack 알림: 완료
  - 인앱 알림: 구현 중 (잔여 약 1일)
잔여 작업: 인앱 알림 구현 + 테스트 코드 작성
차단 요소: 없음
전망: 기한 (2/20)에 맞출 수 있습니다.""",
    intent="report"
)
```

## 일일 요약 포맷

일일 요약은 업무 종료 시 상사에게 보내는 하루 되돌아보기입니다.

### 템플릿

```
send_message(
    to="manager",
    content="""[일일 요약] 2026-02-15

■ 완료
- {완료 태스크 1}
- {완료 태스크 2}

■ 진행 중
- {진행 중 태스크} (진행: {XX%}, 전망: {기한})

■ 과제/우려
- {있으면 기재}

■ 내일 예정
- {예정 1}
- {예정 2}""",
    intent="report"
)
```

### 구체적 예시

```
send_message(
    to="manager",
    content="""[일일 요약] 2026-02-15

■ 완료
- API 사양서 v2.1 갱신 (/shared/docs/api-spec.md)
- 로그 모니터링 스크립트 버그 수정 (타임존 처리)

■ 진행 중
- 사용자 알림 기능 구현 (진행: 70%, 전망: 2/20 완료 예정)

■ 과제/우려
- 외부 API 응답이 느려지는 조짐 있음 (현시점 영향 없음, 계속 모니터링)

■ 내일 예정
- 인앱 알림 구현 완료
- 테스트 코드 작성 착수""",
    intent="report"
)
```

## 긴급 보고 vs 통상 보고 판단

### 긴급 보고 (즉시, MUST)

다음에 해당하는 경우 다른 작업을 중단하더라도 즉시 보고하세요:

- **서비스 중단**: 사용자에게 영향이 나타난 장애
- **데이터 손실/훼손**: 복구 불가능한 상태가 될 가능성
- **보안 인시던트**: 부정 접근, 정보 유출 의심
- **기한 지연 확정**: 이미 약속한 기한에 맞추지 못하는 것이 확실

긴급 보고 메시지에는 MUST: 서두에 "[긴급]"을 붙일 것.
서비스 중단, 데이터 손실, 보안 인시던트 등 사람의 즉시 대응이 필요한 경우 `call_human`도 병용하세요.

```
send_message(
    to="manager",
    content="""[긴급] 운영 데이터베이스 응답 지연

상황: 운영 DB 응답 시간이 평소의 10배 (평균 300ms → 3000ms)로 악화.
영향: Web UI 페이지 로드가 10초 이상 걸리는 상태.
대처 중: 슬로우 쿼리 특정 시작. 10분 이내 속보 예정.""",
    intent="report"
)
```

### 통상 보고 (다음 heartbeat 또는 일일 요약으로)

다음은 긴급성이 낮으므로 모아서 보고해도 됩니다:

- 계획대로 진행 중인 태스크의 중간 경과
- 경미한 문제로 이미 스스로 해결한 것
- 개선 제안이나 아이디어

## 에스컬레이션 판단 플로차트

보고를 에스컬레이션해야 하는지의 판단 절차:

```
1. 자신의 권한 내에서 해결 가능한가?
   → Yes: 직접 대처하고, 완료 후 결과를 보고
   → No: 2단계로

2. 긴급성이 있는가? (서비스 영향, 데이터 손실 위험)
   → Yes: [긴급]으로 즉시 에스컬레이션
   → No: 3단계로

3. 판단의 선택지를 정리할 수 있는가?
   → Yes: 선택지와 추천을 첨부하여 [판단 의뢰]로 보고
   → No: 상황을 있는 그대로 보고하고 상사의 지시를 구함
```

## 보고 시 주의 사항

### 해야 할 것 (MUST/SHOULD)

- MUST: `send_message`의 `intent`에 `"report"`를 지정 (생략하면 오류 발생)
- MUST: 사실과 추측을 구분하여 기술. 추측에는 "가능성이 높다", "아마도" 등을 부여
- MUST: 영향 범위를 명기. "무엇이", "누구에게" 영향을 미치는지
- SHOULD: 시도한 대책과 결과를 포함. 상사가 같은 대책을 제안하는 낭비를 방지
- SHOULD: 다음 액션을 스스로 제안. 지시 대기만 하지 않기

### 피해야 할 것

- 결론을 뒤로 미루고 경위부터 길게 쓰는 것 (결론을 먼저)
- "문제가 발생했습니다"만으로 구체적 정보가 없는 보고
- 여러 다른 주제를 하나의 메시지에 합치는 것 (주제별로 분리)
- 문제를 숨기거나 가볍게 보이게 하는 것 (정확한 상황 전달)
- 동일 run에서 동일 수신자에게 2회 이상 send_message를 보내는 것 (1통까지. 추가는 Board 사용)
