# 유스케이스: 소프트웨어 개발 지원

코드 리뷰, PR 관리, 테스트 실행, 버그 조사 등 소프트웨어 개발 워크플로우를 자동화하고 지원하는 유스케이스입니다.

---

## 이런 과제를 해결할 수 있습니다

- PR이 쌓여 리뷰가 따라잡지 못함
- CI가 실패해도 알아차리는 것이 늦음
- 버그 보고부터 조사 시작까지 시간이 걸림
- 테스트 실행을 잊기 쉬움
- 코드 품질 기준이 속인화되어 있음

---

## 패턴 1: 자동 코드 리뷰

### 하는 일

PR이 생성되면 자동으로 코드를 읽고 문제점이나 개선 제안을 코멘트합니다.

### 동작 흐름

1. 새로운 PR 생성을 감지 (Webhook 또는 정기 체크)
2. 차분을 가져와서 분석
3. 다음 관점에서 리뷰:
   - 보안 문제 (SQL 인젝션, 하드코딩된 비밀 정보 등)
   - 성능 우려
   - 기존 코드와의 일관성
   - 테스트 유무와 커버리지
4. 리뷰 결과를 PR에 코멘트

### 활용 예시

- 신입 엔지니어의 PR에 기본적인 품질 체크를 자동 실행
- 보안 체크리스트를 자동 적용
- 코딩 규약 준수 체크

### 핵심 포인트

- 사람의 리뷰를 "대체"하는 것이 아니라 "보조"하는 포지셔닝
- 자동 리뷰로 기본적인 문제를 처리하고, 사람은 설계 판단에 집중
- 리뷰 기준은 커스터마이즈 가능 (엄격하게/느슨하게)

---

## 패턴 2: CI/CD 결과 감시와 대응

### 하는 일

CI(지속적 통합)의 실행 결과를 감시하고, 실패 시 원인 분석과 수정 제안을 수행합니다.

### 동작 흐름

1. CI 실행 완료를 감지
2. 결과가 실패인 경우:
   - 에러 로그를 가져와 분석
   - 실패 원인을 특정 (테스트 실패, 빌드 에러, 환경 문제 등)
   - 수정안을 작성
   - 담당자에게 알림
3. 결과가 성공인 경우:
   - 머지 조건을 확인
   - 모든 조건을 충족하면 사람에게 승인을 요청

### 활용 예시

- 테스트 실패 시 "이 3개 파일의 변경이 원인입니다. 수정안은 여기입니다"라고 제시
- 환경 의존 실패(flaky test)를 감지하여 자동 재시도
- CI 성공 시 "머지 가능합니다. 변경 요약은 여기입니다"라고 보고

## 패턴 3: Issue 작성부터 구현까지

### 하는 일

버그 보고나 기능 요청에서 Issue 작성 → 브랜치 생성 → 구현 → PR 생성까지 자동화합니다.

### 동작 흐름

1. 사람의 지시를 받아 Issue를 작성 (요건 정의)
2. 작업 브랜치를 자동 생성
3. 지시 내용에 따라 코드를 구현
4. 테스트를 실행하여 동작 확인
5. PR을 작성하고 리뷰를 요청

### 활용 예시

- "로그인 화면에 유효성 검사를 추가해 줘" → Issue 작성 → 구현 → PR 작성까지 자동
- "이 버그를 조사해서 수정해 줘" → 원인 조사 → 수정 → 테스트 → PR 작성

### 주의사항

- 완전 자동 머지는 권장하지 않습니다 (사람의 최종 확인은 필수)
- 대규모 설계 변경은 사람과 상담 후 착수하세요
- 구현 결과는 반드시 테스트로 검증합니다

---

## 패턴 4: 버그 조사 및 원인 분석

### 하는 일

버그 보고를 받아 로그 분석이나 코드 조사를 수행하고, 원인 특정과 수정 방침 제안을 합니다.

### 동작 흐름

1. 버그 보고 내용을 분석 (재현 조건, 영향 범위)
2. 관련 코드를 검색 및 분석
3. 로그와 에러 트레이스를 조사
4. 근본 원인에 대한 가설을 수립
5. 수정 방침과 영향 범위를 보고

### 활용 예시

- "프로덕션에서 ○○이 동작하지 않습니다" → 로그를 분석하여 원인과 수정 방침을 보고
- "최근 성능이 저하되었습니다" → 최근 커밋을 분석하여 원인 후보를 특정
- "테스트가 불안정합니다" → flaky test 패턴을 분석하여 안정화 방안을 제안

---

## 패턴 5: 문서 자동 생성

### 하는 일

코드 변경 내용에서 체인지로그나 릴리스 노트를 자동 생성합니다.

### 동작 흐름

1. 지정 기간의 커밋 이력을 가져옴
2. 변경 내용을 카테고리별로 분류 (신기능/버그 수정/개선 등)
3. 사용자 대상 설명문을 생성
4. 형식에 맞춘 문서를 작성

### 활용 예시

- 릴리스 전에 체인지로그를 자동 생성
- 월간 개발 진행 리포트를 자동 작성
- API 변경의 영향 범위를 정리한 문서를 생성

## 구성 힌트

### 최소 구성 (Anima 1개)

- 1개가 PR 감시, 리뷰, CI 확인을 겸임
- 소규모 프로젝트(월 10 PR 정도)라면 충분

### 권장 구성 (Anima 3~5개)

- **개발 총괄**: 전체 진행 관리, 태스크 배분
- **구현 담당**: Issue 대응, 코드 구현
- **리뷰 담당**: 코드 리뷰, 품질 체크
- **테스트 담당**: 테스트 실행, CI 감시
- 총괄이 태스크를 받아 적절한 담당에게 배분하는 체제

### 품질을 유지하는 요령

- 자동 구현 결과는 반드시 리뷰를 거칩니다 (Anima에 의한 리뷰도 가능)
- 테스트 차분을 확인합니다 ("테스트를 고쳐서 통과시킨" 케이스를 방지)
- main 브랜치에 대한 직접 커밋은 금지하고 PR을 통한 머지를 철저히 합니다
