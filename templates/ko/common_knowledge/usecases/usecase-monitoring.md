# 유스케이스: 인프라 및 서비스 모니터링

서버, 웹 서비스, 클라우드 리소스 등을 24시간 365일 감시하고, 이상 감지, 알림, 초동 대응을 자동화하는 유스케이스입니다.

---

## 이런 과제를 해결할 수 있습니다

- 서버 다운을 알아차리는 것이 늦음
- 야간/휴일의 장애 대응을 할 수 없음
- SSL 인증서 만료를 놓침
- 디스크 용량이나 리소스 고갈을 알아차리지 못함
- 모니터링 도구의 알림이 너무 많아 중요한 것을 놓침

---

## 패턴 1: 웹 서비스 가동 감시

### 하는 일

웹 서비스가 정상적으로 응답하는지 정기적으로 체크하고, 이상 시 알림을 발송합니다.

### 동작 흐름

1. 정기 간격(예: 5~30분마다)으로 대상 URL에 접속
2. HTTP 응답 코드 및 응답 시간을 확인
3. 이상을 감지한 경우:
   - 1회째: 재확인 (일시적 에러 가능성)
   - 2회 연속: 사람에게 알림
   - 복구를 감지하면 복구 알림을 전송
4. 응답 시간 추이를 기록하고 열화 경향을 감지

### 활용 예시

- 프로덕션 웹사이트가 503을 반환 → 즉시 관리자 스마트폰에 알림
- API 응답이 평소의 3배 느려짐 → 성능 열화 경고
- 유지보수 종료 후 복구 확인 → 정상 복귀를 자동 확인 및 보고

### 시작하기 위해 필요한 것

- 감시 대상 URL 목록
- 알림 수신처 (채팅, 이메일 등)
- 판정 기준 (몇 초 이상이면 지연으로 간주할지 등)

---

## 패턴 2: 서버 리소스 감시

### 하는 일

CPU, 메모리, 디스크 용량, 프로세스 상태 등의 서버 리소스를 감시합니다.

### 동작 흐름

1. 정기적으로 서버 상태를 취득 (SSH 또는 API 경유)
2. 각 지표를 임계값과 비교:
   - 디스크 사용률 > 80% → 경고
   - 디스크 사용률 > 95% → 긴급 알림
   - 메모리 사용률의 급격한 상승 → 메모리 누수 의심
   - CPU 고부하 지속 → 폭주 프로세스 체크
3. 이상이 있으면 알림과 권장 대응을 통지

### 활용 예시

- 디스크 용량이 5% 남음 → 오래된 로그 삭제 후보를 제시
- 특정 프로세스가 CPU 100%에서 고정 → 프로세스명과 시작 시간을 보고
- 메모리 사용량이 날마다 증가 → 누수 가능성을 경고

### 자동 대응 예시

- 로그 로테이션 (오래된 로그 파일의 압축 및 삭제)
- 폭주 프로세스의 특정과 보고 (자동 kill은 사람의 승인 후)
- 임시 파일 정리

---

## 패턴 3: SSL/TLS 인증서 만료 감시

### 하는 일

SSL 인증서의 유효 기한을 정기 체크하고, 만료 전에 갱신을 촉구합니다.

### 동작 흐름

1. 매일 1회 대상 도메인의 인증서 유효 기한을 확인
2. 남은 일수에 따라 알림 레벨을 변경:
   - 30일 남음 → 정보 알림 (갱신 준비를 권장)
   - 14일 남음 → 경고 알림 (빠른 갱신을)
   - 7일 남음 → 긴급 알림 (즉시 대응 필요)
3. 갱신 절차를 기재한 리마인더를 전송

### 활용 예시

- 여러 도메인의 인증서 만료를 일괄 관리
- Let's Encrypt 등의 자동 갱신이 정상 동작하는지 확인
- 갱신 후 인증서가 올바르게 적용되었는지 검증

---

## 패턴 4: 클라우드 서비스 감시

### 하는 일

클라우드상의 리소스(컨테이너, 데이터베이스, 스토리지 등)의 상태와 비용을 감시합니다.

### 동작 흐름

1. 클라우드 API를 사용하여 리소스 상태를 정기 취득
2. 다음을 감시:
   - 서비스 가동 상태 (정상/이상)
   - 에러 로그 발생 상황
   - 비용 추이 (예산 초과 감지)
3. 이상이나 예산 초과를 감지하면 알림

### 활용 예시

- 컨테이너 서비스의 태스크 수가 감소 → 서비스 장애 의심
- 데이터베이스 커넥션 수가 상한에 근접 → 스케일업 제안
- 이번 달 클라우드 비용이 예산의 80%를 초과 → 비용 경고

---

## 패턴 5: 로그 분석과 이상 감지

### 하는 일

애플리케이션 로그를 정기적으로 분석하여 에러 패턴이나 비정상적 경향을 감지합니다.

### 동작 흐름

1. 지정 기간의 로그를 취득
2. 에러 및 경고 발생 건수를 카운트
3. 평상시와의 비교로 이상을 판정:
   - 에러율이 평소의 3배 → 조사 알림
   - 새로운 유형의 에러가 발생 → 신규 장애 의심
   - 특정 엔드포인트에서 에러 집중 → 해당 기능의 장애
4. 분석 결과를 요약으로 보고

### 활용 예시

- "지난 30분간 에러 0건, 정상" → 정상 보고
- "새로운 에러 패턴을 3건 감지" → 상세를 첨부하여 보고
- "특정 API의 에러율이 급상승" → 원인 조사를 개시

---

## 패턴 6: 통합 대시보드형 운용

### 하는 일

여러 감시 결과를 통합하여 정기적으로 요약 리포트를 생성합니다.

### 동작 흐름

1. 각 감시 담당으로부터의 보고를 집약
2. 전체 상태를 한눈에 알 수 있는 요약으로 정리
3. 정기(예: 매일 아침 9시)에 사람에게 보고

### 리포트 예시

```
== 오늘의 인프라 상황 ==
[정상] 웹 서비스: 응답 정상 (평균 120ms)
[정상] 서버 리소스: CPU 15%, 메모리 45%, 디스크 62%
[주의] SSL 인증서: example.com 만료까지 20일 남음
[정상] 클라우드 서비스: 전체 태스크 가동 중
[정상] 에러 로그: 지난 24시간 에러 0건
```

## 구성 힌트

### 최소 구성 (Anima 1개)

- 1개로 전체 항목을 순찰 감시
- 순찰 간격 약 30분
- 소규모 환경(서버 1~2대)이라면 충분

### 권장 구성 (Anima 3~4개)

- **감시 총괄**: 전체 상태 파악, 이상 판단, 에스컬레이션
- **서버 감시**: OS, 프로세스, 리소스 감시
- **네트워크/SSL 감시**: SSL 인증서, DNS, 접속성 감시
- **클라우드 감시**: 클라우드 서비스 고유의 감시

### 오보를 줄이는 요령

- 1회의 이상으로 즉시 알림하지 않습니다 (2회 연속으로 확인)
- 임계값은 실제 환경의 평상값을 기준으로 설정합니다
- 감시 Anima의 보고를 총괄이 독립 검증하는 체제로 합니다
- 오보 패턴을 지식으로 축적하여 판정 정확도를 향상시킵니다
