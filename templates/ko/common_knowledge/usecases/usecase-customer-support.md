# 유스케이스: 고객 지원

고객 문의 대응, 에스컬레이션 관리, 대응 품질 유지를 자동화하는 유스케이스입니다.

---

## 이런 과제를 해결할 수 있습니다

- 문의에 대한 초기 응답이 느림
- 대응 누락이 발생함
- 같은 질문에 반복 답변하고 있음
- 영업 시간 외 문의에 대응할 수 없음
- 대응 이력이 정리되지 않음

---

## 패턴 1: 1차 대응 자동화

### 하는 일

고객으로부터 문의를 수신하면 초기 접수 응답과 분류를 자동으로 수행합니다.

### 동작 흐름

1. 문의 channel(이메일, 채팅, 양식 등)을 감시
2. 새로운 문의를 감지
3. 즉시 접수 확인 메시지를 전송 ("문의해 주셔서 감사합니다. 확인 후 답변드리겠습니다")
4. 문의 내용을 분류:
   - 기술적 질문
   - 요금/계약 관련 질문
   - 버그 보고
   - 요청/피드백
   - 기타
5. 분류에 따라 적절한 대응 흐름으로 배분

### 활용 예시

- 양식에서 문의 → 30초 이내에 접수 확인 이메일 전송
- "로그인할 수 없습니다" → FAQ의 비밀번호 재설정 절차를 안내
- "청구서가 도착하지 않았습니다" → 경리 담당에게 에스컬레이션

### 주의사항

- 자동 응답의 범위는 단계적으로 확대하세요
- 처음에는 "접수 확인 + FAQ 안내"부터 시작하는 것이 안전합니다
- 민감한 내용(불만, 법적 문제)은 반드시 사람에게 전달합니다

---

## 패턴 2: FAQ 기반 자동 답변

### 하는 일

과거 대응 이력이나 FAQ 데이터베이스에서 유사한 질문을 검색하여 답변을 제시합니다.

### 동작 흐름

1. 문의 내용을 분석
2. FAQ 데이터베이스에서 유사 질문을 검색
3. 일치도가 높은 답변이 발견된 경우:
   - 답변안을 작성
   - 신뢰도 높음 → 자동 답변 (사람 확인 없이)
   - 신뢰도 중간 → 사람에게 답변안을 제시 (승인 후 전송)
   - 신뢰도 낮음 → 사람에게 에스컬레이션
4. 답변 결과를 로그에 기록

### 활용 예시

- "요금 플랜을 알려주세요" → FAQ에서 요금표를 자동 답변
- "○○ 기능이 있나요?" → 기능 목록에서 해당 답변을 제시
- "해지 방법을 알려주세요" → 해지 절차를 안내 (단, 해지 방지 제안은 사람이 판단)

---

## 패턴 3: 에스컬레이션 관리

### 하는 일

자동 대응으로 해결할 수 없는 문의를 적절한 담당자에게 에스컬레이션하고 대응 상황을 추적합니다.

### 동작 흐름

1. 자동 대응 범위 밖으로 판단
2. 내용에 따라 에스컬레이션 대상을 결정:
   - 기술적 문제 → 개발 팀
   - 계약/요금 → 영업/경리 팀
   - 불만 → 매니저/사람
3. 에스컬레이션 시 요약과 경위를 첨부
4. 대응 기한을 설정하고 진행 상황을 추적
5. 일정 시간 미대응 시 재알림

### 활용 예시

- "버그를 보고합니다" → 개발 팀에 "○○의 버그 보고. 재현 조건: ○○. 우선도: 중"
- 에스컬레이션 후 2시간 경과 → "이 문의가 미대응입니다"라고 리마인드
- 대응 완료 후 → 고객에게 답변을 전송

---

## 패턴 4: 대응 이력 관리와 분석

### 하는 일

모든 문의와 대응 내용을 기록하고 추세 분석과 서비스 개선에 활용합니다.

### 동작 흐름

1. 전체 문의의 기록을 축적:
   - 수신 일시, 내용, 분류
   - 대응 내용, 대응자
   - 해결까지의 소요 시간
   - 고객 만족도 (취득 가능한 경우)
2. 정기적으로 추세를 분석:
   - 빈출 질문 TOP 10
   - 평균 대응 시간의 추이
   - 미해결 건수
3. 분석 결과를 리포트로 보고

### 활용 예시

- "이번 달 문의는 50건. 최다는 ○○에 관한 질문 (15건)"
- "평균 초기 응답 시간: 5분 (지난달: 30분)"
- "○○ 기능 관련 버그 보고가 증가 추세. 개발 팀과의 공유를 권장"

---

## 패턴 5: 선제적 지원

### 하는 일

고객의 문의를 기다리는 것이 아니라 문제를 선제적으로 감지하고 대응합니다.

### 동작 흐름

1. 서비스 상태를 감시
2. 고객에 영향을 미치는 문제를 감지:
   - 서비스 장애 발생
   - 유지보수 예정
   - 요금 플랜 변경
3. 영향을 받는 고객에게 사전 알림

### 활용 예시

- 서비스 장애 발생 → 영향 범위를 특정 → 해당 고객에게 "현재 ○○에 장애가 발생하고 있습니다. 복구까지 잠시 기다려 주세요"
- 계획 유지보수 → 사전에 "○월 ○일 ○시~○시에 유지보수를 실시합니다"
- 계약 갱신일이 가까운 고객 → "계약 갱신일이 다가오고 있습니다. 갱신 절차는 여기를 참조하세요"

## 구성 힌트

### 최소 구성 (Anima 1개)

- 1개가 지원 전반을 담당
- 접수 응답 + FAQ 자동 답변 + 에스컬레이션
- 월간 문의 50건 이하라면 충분히 대응 가능

### 권장 구성 (Anima 2~3개)

- **프론트 담당**: 문의 접수, 분류, FAQ 답변
- **에스컬레이션 담당**: 사람에게 배분, 진행 추적
- **분석 담당**: 대응 이력 분석, 리포트 생성

### 품질 향상 요령

- 자동 답변은 처음에는 "임시 저장 → 사람 확인 → 전송"의 흐름으로 시작합니다
- 고객 대응 톤(정중함 수준)을 사전에 설정합니다
- 대응 템플릿을 충실히 준비합니다 (인사, 사과, 감사, 마무리 등)
- 자동 답변의 정확성을 정기적으로 사람이 샘플 체크합니다
- 새로운 FAQ 항목이 필요하다고 판단되면 즉시 추가합니다
