# インシデント対応手順

> 作成日: 2026-03-02
> 作成者: Sora（リードエンジニア）
> 目的: Day 2のAPI障害（テナント分離バグ）を教訓にした標準手順
> 参照: 実際の対応タイムライン（13:00〜15:00、2時間で完全復旧）

## 5フェーズ対応フロー

### Phase 1: 検知（目標: 5分以内）

**トリガー:**
- 顧客からの報告（Hina経由）
- 監視アラート（Datadog / PagerDuty）
- Board投稿での異常報告

**初動アクション:**
1. Boardに `⚠️` 付きで障害発生を投稿
2. Kaitoに即時報告（intent: report）
3. 影響範囲の初期推定

**Day 2の実績:**
- 13:00 Hinaから報告受信
- 13:02 Kaitoが全体共有
- 初動: 2分

### Phase 2: 切り分け（目標: 15分以内）

**手順:**
1. エラーログの確認
   ```bash
   tail -f /var/log/pixelforge/api.log | grep ERROR
   ```
2. 関連コードの全数調査
   ```bash
   grep -r 'setTenant\|setTeamId' src/api/
   grep -r 'cache.get\|cache.set' src/api/
   ```
3. 直近のデプロイ・変更の確認
   ```bash
   git log --oneline -20
   ```

**Day 2の実績:**
- 13:05 `grep -r` で全テナント設定箇所を調査
- 13:10 design-cache.ts でキャッシュキーにテナントID欠落を発見
- 切り分け: 10分

### Phase 3: 修正（目標: 30分以内）

**手順:**
1. 修正コード作成
2. ユニットテスト追加
3. ローカルで全テスト実行
4. PR作成 → レビュー依頼 → マージ

**Day 2の実績:**
- 13:15 修正コミット + テスト追加
- 13:25 PR #11 作成
- 13:30 マージ
- 修正: 20分

### Phase 4: 通知（目標: 修正後30分以内）

**Hinaへの引き渡しテンプレート:**

```
【Hinaへの依頼】
障害の概要: {1行で}
根本原因: {技術的だが顧客に伝わる表現で}
修正状況: {修正済み / デプロイ済み}
影響: {データ漏洩・変更・削除の有無}
顧客への通知文言案:
  「{事象}が発生しましたが、{修正済み}です。{影響}は確認されていません。」
```

**Day 2の実績:**
- 13:30 Kaitoに修正完了報告 + Hinaへの通知依頼
- 13:35 Kaitoが Hinaに通知を委任
- 15:00 Hina が影響5社に通知完了、全社受領確認
- 通知: 1時間30分（5社分）

### Phase 5: 振り返り（目標: 当日中）

**記録項目:**
1. タイムライン（検知〜復旧の全ステップ）
2. 根本原因分析（5 Whys）
3. 再発防止策
4. ナレッジ更新

**Day 2の実績:**
- `knowledge/tenant-isolation-design.md` を作成（教訓含む）
- PRレビューチェックリストに「キャッシュキーにテナントID」を追加
- E2Eテストにテナント分離テストを追加

## 重要度分類

| レベル | 定義 | 対応時間 |
|--------|------|---------|
| P0 (Critical) | データ漏洩、全面停止 | 即時 |
| P1 (High) | 一部機能停止、データ不整合 | 1時間以内 |
| P2 (Medium) | パフォーマンス劣化 | 4時間以内 |
| P3 (Low) | UI不具合、表示崩れ | 翌営業日 |

## Sora→Hina 連携パターン

障害対応での役割分担:

```
Sora: 検知 → 切り分け → 修正 → Kaitoに報告
       ↓
Kaito: 判断 → Hinaに通知を委任
       ↓
Hina: 影響顧客特定 → 通知送信 → 受領確認追跡
```

**ポイント:**
- SoraはKaitoへの報告に「Hinaへの通知文言案」を含める
- Hinaは `knowledge/customer-communication-patterns.md` のテンプレートを使う
- 通知完了後、HinaはKaitoに報告（全社受領確認含む）
