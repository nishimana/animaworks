# PixelForge リリース管理チェックリスト

> 作成日: 2026-03-04
> 作成者: Kaito（プロダクトマネージャー）
> 目的: v0.3リリースの経験を元にした標準手順

## リリース判断基準

### MUST（リリースブロッカー）
1. **セキュリティ**: CRITICAL/HIGHのオープンIssueがゼロ
2. **テスト**: E2Eテスト全パス、カバレッジ80%以上
3. **パフォーマンス**: APIレスポンス p95 < 200ms
4. **データ**: テナント分離テスト全パス

### SHOULD（可能な限り対応）
1. CHANGELOGの全セクション記載
2. ドキュメント更新
3. 顧客通知テンプレート準備

## リリース手順

### Phase 1: リリース前日
- [ ] Soraにセキュリティ監査を依頼
- [ ] Hinaに顧客通知テンプレート準備を依頼
- [ ] 全オープンPRの確認・マージ判断
- [ ] ステージング環境でのスモークテスト

### Phase 2: リリース当日
- [ ] Soraからセキュリティ監査結果を受領
- [ ] 全テスト結果確認（E2E + ユニット）
- [ ] CHANGELOG最終確認
- [ ] リリースタグ作成（Soraに指示）
- [ ] Hinaに顧客通知送信を指示
- [ ] CEOに完了報告

## v0.3での教訓
- セキュリティ監査は前日に依頼しないと間に合わない（v0.2では当日で混乱）
- 顧客通知はバグ修正内容も含めると反応が良い
- ステージング環境のスモークテストは省略しない
